22. Chained Lookup OOM Stress

This chapter is the on-device acceptance test for the dictionary RAM rework (one-page streaming render, pooled page, compact lookup chain). The chain-stress dictionary holds five deliberately huge, style-dense definitions (~22 KB of text, ~30 pages each, many style changes per line) whose text cyclically names the next headword, so you can chain lookups arbitrarily deep. The goal: free heap stays flat as you chain, and nothing crashes — previously a repeated deep render fragmented the heap and aborted.

Dictionary

chain-stress

Note

Watch free heap over the serial monitor (e.g. python3 scripts/debugging_monitor.py) while running these tests. Heap should return to roughly the same level after each lookup and after backing out — it must not trend downward as the chain deepens.

Each definition marks two selectable words with ** on both sides: the CHAIN word (the next entry, also repeated on every page) and a BRANCH word (a different entry, used in Test C). The ** are just visual anchors — select the word between them to look it up.

Important: these tests build a chain by looking up the next word from within a definition (via Look Up Word). If you instead look a word up directly from this chapter, that definition is the start of a new chain, so pressing Back returns to the word-select screen — that is expected, not a bug. Always start a chain test by looking up chain_stress_00001 below.

Test A: Deep deterministic chain (heap flat, no crash)

Each definition names the next headword on every page, so the chain is deterministic. The first render and every subsequent one must hold only a single page in RAM.

Steps

  1. Look up chain_stress_00001. Definition 1 appears (page 1 of ~30). No crash.
  2. Note the free heap reading.
  3. Press Look Up Word, select chain_stress_00002 (it appears mid-line on every page), Confirm. Definition 2 appears.
  4. Repeat: from definition 2 look up chain_stress_00003, then chain_stress_00004, then chain_stress_00005. Five definitions deep, no crash.
  5. Verify free heap after definition 5 is within a small margin of the reading from step 2 — it must not have dropped by tens of KB per hop.

Test B: Back-navigation titles and page restore

Backing out of the chain must return to each prior definition with the same title you saw going forward, on the page you were reading.

Steps

  1. From definition 1, page forward to about page 5 (use PageForward).
  2. Press Look Up Word, select chain_stress_00002, Confirm. Definition 2 appears at page 1.
  3. Short-press Back. You return to definition 1 with title chain_stress_00001, on page 5 (not page 1).
  4. Short-press Back again. You exit to the reader (chain is empty). No crash.

Test C: Back-then-forward (abandoned branch not revisited)

Chaining forward after backing up must not pull the abandoned branch back into the stack.

Steps

  1. Look up chain_stress_00001 (A). Look Up Word → select its CHAIN word chain_stress_00002 (B). From B, Look Up Word → select B's CHAIN word chain_stress_00003 (C).
  2. Short-press Back once — you return to B (chain_stress_00002).
  3. From B, Look Up Word → select B's BRANCH word chain_stress_00004 (D) instead (marked ** ... ** near the top of B).
  4. Short-press Back. You return to B, not the abandoned C.
  5. Short-press Back again. You return to A (chain_stress_00001). The abandoned C is never revisited.

Test D: Deep chain past the history limit (eviction)

Because the definitions cycle (00005 names 00001 again), you can chain well past the Dictionary History Limit. The chain depth is bounded by that limit; going deeper must stay stable, not crash or leak.

Steps

  1. Set the Dictionary History Limit to its minimum in settings.
  2. Look up chain_stress_00001 and chain forward continuously (each definition names the next), at least 30 hops deep. Heap stays flat throughout.
  3. Back out repeatedly. The earliest entries may exit to the reader once the history window is exceeded; this is expected. No crash at any depth.
